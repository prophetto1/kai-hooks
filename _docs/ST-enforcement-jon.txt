What I do not want for st enforcement:
  - once per user prompt only,
  - once per entire long task only,
  - before every tool call,
  - or as a mechanical “pay the toll before Bash” loop.

Rather, what we want:

  The clearest articulation is:

  > A worker doing a long task should be required to re-enter deliberate planning at major phase transitions, not after every action and not only at the beginning.

BUT: a hook fires on events (PreToolUse, Stop…), and "phase transition" is not an event — it's semantic. So the entire design problem reduces to: turn "major phase transition" into a signal a deterministic hook can detect. Everything else follows.

YET is there methods that either external trigger e.g. an implementation plan already drafted. Requirements could be things like (these are not actually set in stone but examples) :
- should create a grant/window.
- That grant should cover a coherent batch of related actions.
- The grant should expire by time, tool count, or phase transition.
- The agent should need a new checkpoint when the work materially changes.
- Higher-risk actions should force a new checkpoint earlier.


ON THE OTHER HAND:
For internal trigger e.g. certain type of tools that would indicate a chain-of-thought or some form of internal process has completed and a new one is starting.

NONETHELESS: The mechanism that fits: a bounded grant that re-arms at some type of deterministically calculatable boundary.
 1. LangGraph checkpoints / interrupts
     LangGraph persists state at graph step boundaries and supports interrupt/resume. This is the cleanest analogy for “major phase transition becomes a
     durable boundary.”

  2. Temporal / durable execution
     The 2026 trend is durable, resumable agent workflows. The useful idea is not “use Temporal here,” but: store workflow state, replay from checkpoints,
     and make execution boundaries explicit.

  3. Claude Code hooks
     Claude’s current hook system explicitly supports deterministic lifecycle controls: PreToolUse, PostToolUse, PostToolBatch, Stop, allow/deny/ask/
     defer, and pre-approved safe permissions. This supports your exact constraint: hooks fire often, but only disrupt when a deterministic boundary says
     they must.

  4. Recent research trend
     2026 papers are moving toward guardrails over multi-step tool trajectories, not just final answers. “Guardrails as Infrastructure” and “TraceSafe”
     both reinforce that agent safety/control must inspect structured tool traces and workflow state.
     Sources:
     https://arxiv.org/abs/2603.18059 --- literature review required
     https://arxiv.org/abs/2604.07223 --- literature review required


----------
how to reconcile?

  - Keep bounded grants, but issue/re-arm them automatically.
  - Do not require manual checkpoint commands as normal behavior.
  - Treat “plan token” as an internal audit/grant record, not a new agent habit.
  - Detect phase changes from deterministic signals: tool-class shift, edit batch start, verification start, repeated failure, protected write, scope/path
    change, Stop/finalization.
  - External sequentialthinking becomes optional aid, not the authorization primitive.
