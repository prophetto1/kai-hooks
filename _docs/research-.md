# Deterministic Phase-Boundary Detection and Bounded Auto-Re-Arming Grants for Agent Tool-Use Trajectories: A Design-Oriented Research Brief

## TL;DR
- **A "major phase transition" can be turned into a deterministic, hook-observable signal** by computing it from the structured tool-trajectory state (tool_name + collapsed target + status + timestamp + session_id). The single most reliable, non-gameable arming signal is the **operation-mode transition** — specifically the *first state-changing action after a run of read-only actions* (explore→mutate) and the *first verification action after a run of edits* (mutate→verify) — both computable from tool class alone and not under the agent's narrative control.
- **The grant must be an internal audit/grant record the gate writes automatically**, never an agent-authored token. Prior art strongly supports this: Progent, AgentSpec, and the AEGIS/behavioral-firewall line all enforce *deterministic, model-independent* gating at the tool-call boundary, and the durable-execution frameworks (LangGraph, Temporal, DBOS) all define boundaries *structurally* (super-steps, journaled events) rather than letting the workload declare them.
- **Both papers the user cited resolve as real.** arXiv:2604.07223 is *TraceSafe* (CyCraft AI Lab / NTU) and arXiv:2603.18059 is *Guardrails as Infrastructure: Policy-First Control for Tool-Orchestrated Workflows* (Sigdel & Baral). The Claude Code hook surface the user cited is also real — **PostToolBatch exists**, and the PreToolUse decision vocabulary is **allow / deny / ask**, plus **defer** (non-interactive `-p` mode only). Codex CLI hooks exist but currently only intercept `Bash`.

## Details

### A) Durable execution / checkpoint boundaries — the transferable concept

The durable-execution ecosystem provides the conceptual spine for "a phase boundary = a durable, re-armable grant." The key insight, repeated across every framework, is that **execution boundaries are defined structurally by the runtime, not declared by the workload.**

- **LangGraph** creates a checkpoint at each **super-step boundary**. Per LangChain's docs (verbatim): "LangGraph creates a checkpoint at each super-step boundary. A super-step is a single 'tick' of the graph where all nodes scheduled for that step execute (potentially in parallel)." A checkpoint is a `StateSnapshot` persisted by a checkpointer and keyed by `thread_id`. Crucially for our design, the boundary is a structural property of the graph's execution computed by the runtime. Human-in-the-loop is implemented via `interrupt()`, which "saves the graph state using its persistence layer and waits indefinitely until you resume." Resumption uses `Command(resume=...)` against the same `thread_id`. The reusable idea: **a boundary is durable because state is snapshotted there and is addressable by a stable pointer.** One important warning LangGraph documents: on resume, the node *restarts from the beginning*, so side effects before an interrupt must be idempotent — directly relevant to writing a grant record exactly once.
- **Temporal** models durability via an immutable **event history**. Per Temporal's official docs (verbatim): "if the Worker crashes, the Worker uses the Event History to replay the code and recreate the state of the Workflow Execution to what it was immediately before the crash," and "If a failure occurs, the Workflow Execution picks up where the last recorded event occurred in the Event History." The transferable pattern (NOT "adopt Temporal"): record progress as an append-only log of completed events, and make execution boundaries explicit and replayable. Note the critical determinism constraint Temporal documents: non-deterministic work (LLM calls) must be wrapped as journaled "activities" so they are not re-executed on replay.
- **DBOS** checkpoints each step's output into Postgres; on restart it "resumes all incomplete workflows from the last completed step" by re-executing the deterministic workflow function and skipping already-journaled steps. DBOS makes the "save point" metaphor explicit and provides a native OpenAI Agents SDK integration.
- **Restate** uses the same journal/replay model, lighter-weight than Temporal and better suited to serverless/edge.

The synthesis across all of them is well stated by The Agent Stack newsletter (Substack), "Part 4: Runtimes, Workflows, and Durable Execution" (verbatim): "A workflow gives that run a recoverable shape. Durable execution records enough progress for the work to survive time, failure, waiting, and retry," and "Durable execution is recorded progress before the world becomes ambiguous." For our control plane, the grant record is exactly this: a durable, append-only audit entry written *by the gate* at a deterministically computed boundary.

### B) Trajectory-level guardrails research (2025–2026)

The clear research trend is that the vulnerability surface has moved from final outputs to **intermediate execution traces**, and guardrails must operate over multi-step tool trajectories.

- **TraceSafe (arXiv:2604.07223).** Introduces TraceSafe-Bench, "the first comprehensive benchmark specifically designed to assess mid-trajectory safety," with 12 risk categories and 1,000+ execution instances, built via a "Benign-to-Harmful Editing" methodology that *deterministically injects targeted risks into natural trajectories … providing precise, step-level ground truth labels.* Three findings directly relevant to our design: (1) **Structural Bottleneck** — guardrail efficacy is driven more by structural data competence (e.g., JSON parsing) than semantic safety alignment (ρ=0.79 with structured-to-text benchmarks, near-zero with jailbreak robustness); (2) **Architecture over Scale**; (3) **Temporal Stability** — accuracy is resilient across extended trajectories, and *later* steps actually improve risk detection. Takeaway: a deterministic, *structural* gate over the trace is well-aligned with what actually drives guardrail performance.
- **Guardrails as Infrastructure (arXiv:2603.18059).** Proposes "Policy-First Tooling, a model-agnostic permission layer that mediates tool invocation through explicit constraints, risk-aware gating, recovery controls, and auditable explanations." It contributes a compact policy DSL and a runtime enforcement architecture. Quantitatively, across 225 controlled runs, stricter policy packs raised violation prevention from 0.000 (P0) to 0.681 (P4) while task success dropped from 0.356 to 0.067, and **retry amplification dropped from 3.774 to 1.378** under stricter controls. This is the central thesis we adopt: *"guardrails can be engineered, measured, and tuned as infrastructure rather than treated as model behavior."*
- **AEGIS (arXiv:2603.12621).** "No Tool Call Left Unchecked — A Pre-Execution Firewall and Audit Layer for AI Agents" (Aojie Yuan, USC; Zhiyuan Su, UC Davis; Yue Zhao, USC). A *stateless, single-call* pre-execution firewall with a three-stage pipeline (deep string extraction to depth 32 → content scanning against 22 patterns in 7 categories → cached JSON-Schema policy validation). It returns one of three decisions — **allow (LOW/MEDIUM), block (policy violation), or pending (HIGH/CRITICAL, routed to human review)** — which maps directly onto Claude Code's allow/deny/ask. It blocked all attacks in a curated 48-instance suite, adds 8.3 ms median latency (P95 14.7 ms, P99 23.1 ms) across 1,000 interceptions, with a 1.2% false-positive rate on 500 benign calls. It explicitly contrasts with observability platforms (Langfuse/Helicone/Arize): *"these platforms improve visibility after a tool call has been issued or executed, but they do not provide a framework-agnostic pre-execution enforcement layer that can block or escalate calls before side effects occur."*
- **"Enforcing Benign Trajectories: A Behavioral Firewall for Structured-Workflow AI Agents" (arXiv:2604.26274, Hung Dang, Van Lang University).** The genuinely *sequence-level* artifact. It "compiles verified benign tool-call telemetry into a parameterized deterministic finite automaton (pDFA)" — each state is a tool name plus the last 3 calls, with learned per-transition parameter bounds — then enforces it at runtime "via an O(1) state-transition structural lookup," with offline profiling from ~400 clean traces. Reported attack-success-rate of **2.2% vs. 12.8% for AEGIS** (stateless) on three structured workflows, at 2.2 ms median per-call latency. Two cautions pivotal for our design: (1) it **breaks 24.0% of benign tasks ("24.0% BTFR") when 20% of the tool set changes** (concept drift), restored to 3.8% after one incremental update cycle (350 human-approved sequences, ~8.75 analyst-hours); (2) it degrades badly on *less-structured* agents (12.6% ASR on a Research Agent, 8.6% on a Travel Planner). Lesson: learned sequence-shape enforcement is powerful but brittle to drift — argues for *coarse, stable* signals (tool *class* transitions) rather than fine-grained learned sequences for the re-arm primitive. (Note: "Praetor" is the nickname used only in a secondary blog by Ilya Kabanov at theweatherreport.ai; the paper itself uses an anonymized codename.)
- **Progent (arXiv:2504.11703).** Authors: Tianneng Shi, Jingxuan He, Zhun Wang, Linyu Wu, Hongwei Li, Wenbo Guo, Dawn Song. The latest version (v3, 14 May 2026) is retitled "Progent: Securing AI Agents with Privilege Control." It is the first privilege-control mechanism for LLM agents, using a DSL over tool names and arguments (verbatim): "Every tool call is checked against such a policy through a deterministic procedure … Each proposed update is determined by an SMT solver to be either a narrowing (applied automatically) or an expansion." Evaluated on AgentDojo, ASB, and AgentPoison.
- **AgentSpec (arXiv:2503.18666v3, 31 Jul 2025; ICSE'26).** Authors: Haoyu Wang, Christopher M. Poskitt, Jun Sun. A lightweight DSL with a trigger/predicate/enforcement rule structure ("a rule may enforce human verification before an agent modifies sensitive data, or enforce self-reflection via an LLM before an agent proceeds with a high-risk task"). Abstract (verbatim): "AgentSpec successfully prevents unsafe executions in over 90% of code agent cases, eliminates all hazardous actions in embodied agent tasks, and enforces 100% compliance by autonomous vehicles (AVs) … with overheads in milliseconds." LLM-generated rules (OpenAI o1, few-shot) attained 95.56% precision and 70.96% recall for embodied agents. Both Progent and AgentSpec validate the core design choice: gating logic must live *outside* the agent and be deterministic.

### C) Trajectory segmentation / phase detection / replan triggers

- **ReAct (arXiv:2210.03629).** Interleaves reasoning, acting, and observation, where "reasoning traces help the model induce, track, and update action plans … while actions allow it to interface with external sources." Confirmed: **ReAct does NOT prescribe forced reasoning before every action** — it is a flexible thought-action-observation interleaving, and the paper notes "more examples do not improve performance" and even backs off to CoT-SC when ReAct fails. This directly supports the user's framing that deliberate planning should be re-entered at major transitions, not before every action.
- **Segmentation is normally done semantically by an LLM, not deterministically.** "Trajectory-Informed Memory Generation" (arXiv:2603.10600) has an LLM segment a full trajectory into subtasks by "identify[ing] natural boundaries between distinct logical phases — transitions from authentication to data retrieval, from data retrieval to processing." GUIDE (arXiv:2604.04399) does hierarchical LLM segmentation for GUI-agent evaluation. TrajAD (arXiv:2602.06443) does step-level anomaly detection and localization, halting before step t+1 when an anomaly is detected and rolling back to t−1. **The key gap our design fills:** these are semantic, model-based segmenters; the user needs a *deterministic* proxy. The tool-class transition is exactly that proxy — it approximates "authentication→retrieval→processing" boundaries using the operation-mode signature alone.
- **Replan triggers (classical + LLM).** Classical execution monitoring ("Approximately Optimal Monitoring of Plan Preconditions," arXiv:1301.3839) defines replanning triggers as **precondition failure, environment divergence from the expected state, or goal change.** "Scene Graph-Guided Proactive Replanning" (arXiv:2508.11286) triggers replanning at **subtask boundaries** when the observed state diverges from the expected precondition state. The "Scheduler-Theoretic Framework" (arXiv:2604.11378) is especially relevant: it argues recovery actions must have "explicit trigger conditions, bounded scope, and a strict escalation protocol," found that 70 analyzed agent systems "commonly lacked any formal bounds on recovery attempts" leading to "infinite retry loops" or "premature abandonment," and enforces a per-node recovery counter with a strict escalation order (retry → patch → replan). **Reflexion (arXiv:2303.11366)** operationalizes the failure-driven re-plan (verbatim): "we allowed the agent to gather experience and retry on failed tasks until it produced 3 consecutive failed attempts on the particular task," with self-reflection memory "truncate[d] … to the last 3 self-reflections." This is the canonical "N consecutive same-error failures forces a re-plan" pattern.
- **Plan-and-Execute / hierarchical replanning.** LangChain's Plan-and-Execute and the "Why Do LLM-based Web Agents Fail?" hierarchical-planning study (arXiv:2603.14248) both use a **postcondition checker** after a subgoal's low-level actions; replanning is triggered only when the postcondition fails after multiple iterations. This validates verification-start as a natural phase boundary, and localized replanning (only the affected subtree) as the efficient response.

### D) Primary-source hook & tool-gating capabilities (verified against current docs)

**Claude Code hooks** (code.claude.com/docs/en/hooks-guide):
- **Events that exist and are relevant:** SessionStart, UserPromptSubmit, **PreToolUse** ("Before a tool call executes. Can block it"), **PermissionRequest**, **PermissionDenied** (`{retry: true}` to allow retry), **PostToolUse** ("After a tool call succeeds"), **PostToolUseFailure** ("After a tool call fails"), **PostToolBatch** ("After a full batch of parallel tool calls resolves, before the next model call"), **Stop**, **StopFailure**, SubagentStart/SubagentStop, **CwdChanged** ("When the working directory changes, for example when Claude executes a `cd` command"), FileChanged, SessionEnd, and others.
- **PostToolBatch is REAL.** It fires after a batch of parallel tool calls resolves, before the next model call, and has **no matcher support** (always fires).
- **Permission decision vocabulary (PreToolUse):** `"allow"` (skip interactive prompt; deny/ask rules still apply), `"deny"` (cancel + send reason to Claude), `"ask"` (show prompt). A **fourth value, `"defer"`, exists but only in non-interactive `-p` mode** — "exits the process with the tool call preserved so an Agent SDK wrapper can collect input and resume." So the user's cited "allow/deny/ask/defer" is correct, with the caveat that defer is `-p`-only.
- **JSON contract confirmed:** `hookSpecificOutput` with `hookEventName`, `permissionDecision`, `permissionDecisionReason`, `additionalContext`, and `updatedInput`. Exit code 2 = block with stderr fed back to Claude. Input includes `session_id`, `cwd`, `hook_event_name`, `tool_name`, `tool_input`.
- **Critical enforcement property:** "A hook that returns `permissionDecision: \"deny\"` blocks the tool even in `bypassPermissions` mode or with `--dangerously-skip-permissions`." This is what makes a hook-based gate non-bypassable by the agent. Conversely, `"allow"` cannot loosen deny rules — hooks can only tighten.
- **Auto-approval pattern is real:** a `PermissionRequest` hook returning `{"decision": {"behavior": "allow"}}` auto-approves (the docs' ExitPlanMode example), and can set `updatedPermissions` with `setMode` (default/acceptEdits/bypassPermissions).
- **Filtering:** `matcher` filters by tool name; the newer **`if` field** (v2.1.85+) uses permission-rule syntax like `"Bash(git *)"` or `"Edit(*.ts)"` to filter on tool name + arguments together. `if` only works on tool events.
- **Stop hook caveats:** Stop fires "whenever Claude finishes responding, not only at task completion," does not fire on user interrupts, and has a block cap (overridden after 8 consecutive blocks; parse `stop_hook_active` to avoid loops).

**Codex CLI hooks** (developers.openai.com/codex/hooks):
- Hooks exist behind a feature flag (`[features] codex_hooks = true`), **experimental, Windows temporarily disabled.** Events: SessionStart, PreToolUse, PostToolUse, UserPromptSubmit, Stop (PreToolUse/PostToolUse/UserPromptSubmit/Stop are turn-scoped).
- **Major limitation:** PreToolUse and PostToolUse currently **only emit `Bash`** — they do not intercept Edit, Write, MCP, or WebSearch tool calls. The docs explicitly warn: "The model can still work around this by writing its own script to disk and then running that script with Bash, so treat this as a useful guardrail rather than a complete enforcement boundary."
- PreToolUse supports blocking via `{"hookSpecificOutput": {"permissionDecision": "deny", ...}}` or `{"decision": "block"}` or exit 2. `permissionDecision: "allow"`/`"ask"` are "parsed but not supported yet, so they fail open."
- Stop supports `{"decision": "block", "reason": "..."}` to continue (creates a new continuation prompt from `reason`).
- Codex also has an independent **approval policy** (`untrusted`/`on-request`/`never`), **sandbox modes** (read-only / workspace-write / danger-full-access), and protected read-only paths (`.git`, `.codex`, `.agents`). Enterprise `requirements.toml` can force approval/sandbox constraints. The `notify` program currently fires only on `agent-turn-complete`.

**OpenAI Agents SDK HITL** (openai.github.io/openai-agents-python/human_in_the_loop):
- `needs_approval=True` (or an async predicate over parsed params) on `function_tool`, `Agent.as_tool`, `ShellTool`, `ApplyPatchTool`, and MCP servers. When triggered, the run pauses and `RunResult.interruptions` contains `ToolApprovalItem` entries.
- **Durable interruptions:** `result.to_state()` → `state.approve(...)`/`state.reject(...)` → resume with `Runner.run(agent, state)`. `RunState` is serializable (`to_json`/`to_string`) and "designed to be durable" — store pending work in a DB/queue and recreate later. This is the SDK analog of LangGraph's interrupt/resume and exactly the "approvals as durable interruptions" + "validation next to the side-effecting tool" pattern the user wants.

**OpenAI reasoning guidance:** OpenAI's reasoning guidance advises giving clear goals/constraints and not forcing every intermediate reasoning step, consistent with re-entering deliberate planning only at major transitions.

### E) Synthesis — concrete deterministic design for a bounded auto-re-arming grant

**The grant record (internal, gate-authored).** The gate maintains, per `session_id`, a small durable state record: `{phase, mode, grant_armed (bool), grant_id, armed_at_ts, armed_at_toolcount, tool_count, consecutive_failures, last_error_signature, working_set (recent dir-prefixes), last_state_change_target}`. This is the audit/grant record — written *only* by the gate, never by the agent. SequentialThinking (or any MCP reasoning tool) is downgraded to an optional aid the agent *may* call when a grant is armed; calling it is neither necessary nor sufficient to arm a grant.

**Per-signal analysis (1–8):**

**1. Tool-class / operation-mode shift (explore/read → mutate/edit → verify/test → commit).**
- *Deterministic?* **Yes**, from `tool_name` alone via a static classifier map (Read/Grep/Glob/WebFetch=READ; Edit/Write/MultiEdit/apply_patch=MUTATE; Bash matching test/build/lint=VERIFY; Bash matching `git commit`/`git push`=COMMIT). The boundaries "first MUTATE after ≥1 READ" and "first VERIFY after ≥1 MUTATE" are trivially computable on a minimal schema.
- *Hook:* PreToolUse (observe the about-to-run class vs. the running mode in the grant record).
- *Prior art:* Deterministic proxy for the LLM-based "authentication→retrieval→processing" segmentation (arXiv:2603.10600); aligns with permission-tier systems (read=implicit allow, write=scoped, destructive=approval).
- *Gameable?* Low. The agent cannot relabel a tool's class; class derives from the tool name the runtime reports. Edge case: mutations *through* `Bash` (heredocs, `sed -i`) — mitigated by classifying Bash subcommands via the `if` field and a Stop-hook working-tree scan (Claude Code docs explicitly recommend this).
- *Grant action:* **ARM** on READ→MUTATE and on MUTATE→VERIFY transitions.

**2. Edit-batch start/end.**
- *Deterministic?* Partially. Batch *start* = first MUTATE after non-MUTATE (same as signal 1). Batch *end* is cleanly observable on Claude Code via **PostToolBatch** or a MUTATE→(READ|VERIFY) transition. On Codex, no batch event exists.
- *Hook:* PreToolUse (start), PostToolBatch (end, Claude Code only).
- *Prior art:* LangGraph super-step (a "tick" where parallel nodes resolve together) is the conceptual model — PostToolBatch is Claude Code's super-step boundary.
- *Gameable?* Low for start; batch-end timing can be influenced by call grouping, but that does not let the agent avoid the START arm.
- *Grant action:* **ARM** at batch start; **CONSUME** at batch end (a grant authorizes one coherent edit batch).

**3. Verification start (tests/build/lint after edits).**
- *Deterministic?* **Yes** — VERIFY-class Bash command immediately following MUTATE activity.
- *Hook:* PreToolUse (Bash matcher + `if` rule for test/build/lint commands).
- *Prior art:* Plan-and-Execute postcondition checkers (arXiv:2603.14248) treat verification as the subgoal-completion boundary.
- *Gameable?* Low-moderate (agent could skip verification entirely) — handle by also using Stop to require verification before finalize.
- *Grant action:* **ARM** (verify is a new phase) and **CONSUME** the prior edit grant.

**4. Repeated failure / retry loop (Reflexion-style N consecutive same-error).**
- *Deterministic?* **Yes**, if telemetry includes `status` and an error signature. Increment `consecutive_failures` when `status=failure` and the collapsed target/error matches the previous; reset on success or target change.
- *Hook:* PostToolUseFailure (Claude Code); Codex PostToolUse fires on non-zero exit too.
- *Prior art:* Reflexion's 3-consecutive-failures→reflect (arXiv:2303.11366); the Scheduler-Theoretic framework's bounded recovery counter (arXiv:2604.11378) warns explicitly against unbounded retry loops.
- *Gameable?* Low — failures are environment-reported. The agent could vary commands to dodge the "same-error" test; counter this by keying on a coarse error signature (e.g., exit code + collapsed target) rather than exact stderr.
- *Grant action:* **FORCE EARLY RE-ARM** at N consecutive same-error failures (require re-entering deliberate planning); **VOID** any in-flight edit grant.

**5. Protected-path / destructive action.**
- *Deterministic?* **Yes**, by matching the collapsed target (`file_path`/`command`/`path`) against a protected-pattern list (`.env`, `.git/`, `package-lock.json`, prod configs, `rm -rf`, `git push --force`, `DROP TABLE`).
- *Hook:* PreToolUse (with `if` rules like `Edit(*.env)`, `Bash(git push *)`).
- *Prior art:* Claude Code's documented protect-files pattern; AgentSpec's "human verification before modifying sensitive data"; AEGIS's HIGH/CRITICAL→pending routing; Codex's read-only protected paths.
- *Gameable?* Low if enforced via `permissionDecision: "deny"` (non-bypassable even under `--dangerously-skip-permissions`).
- *Grant action:* **FORCE EARLY RE-ARM / escalate to `ask`** — a destructive action requires a fresh, stronger grant regardless of remaining budget; never consumable by a routine grant.

**6. Scope / path change / working-set drift.**
- *Deterministic?* **Yes but needs slightly richer telemetry.** Compute the directory-prefix of each edit target; maintain a working set of recent prefixes; flag a boundary when a new edit's top-level dir-prefix is outside the current cluster (or the Jaccard distance of the working set exceeds a threshold). Claude Code's **CwdChanged** event gives a free, fully deterministic drift signal when the agent runs `cd`.
- *Hook:* PreToolUse (compute prefix from target) or CwdChanged (Claude Code).
- *Prior art:* The behavioral-firewall pDFA (arXiv:2604.26274) tracks working-set shape but is brittle to drift (24% benign breakage at 20% tool change) — argues for a *coarse* dir-prefix test, not a learned model.
- *Gameable?* Moderate — the agent controls which files it edits, but cannot hide the target from the hook. Keep the threshold coarse to avoid false re-arms.
- *Grant action:* **ARM** on cluster change (a new working-set cluster = a plausibly new subtask).

**7. Stop / finalization / turn boundaries.**
- *Deterministic?* **Yes.** Stop fires when Claude finishes responding.
- *Hook:* Stop / StopFailure (Claude Code); Stop (Codex).
- *Prior art:* Durable-execution "turn boundary" and OpenAI Agents SDK interruption-at-turn-end.
- *Gameable?* Stop fires on every response end, not only task completion (documented), so don't treat Stop alone as task-done; pair with a completeness check. Watch the 8-block cap.
- *Grant action:* **VOID** all in-flight grants at finalization (clean slate); optionally **FORCE RE-ARM** by blocking Stop until verification ran.

**8. TTL and tool-count budgets.**
- *Deterministic?* **Yes**, from `timestamp` and a running `tool_count` in the grant record.
- *Hook:* any hook (checked on every PreToolUse).
- *Prior art:* The Scheduler-Theoretic framework's hard "Max turns" limits; Guardrails-as-Infrastructure's retry-amplification control.
- *Gameable?* Not gameable (clock and counter are gate-owned).
- *Grant action:* **VOID / expiry** — a grant auto-expires after K tool calls or T seconds, forcing re-arm. This is the "auto-re-arm" backstop that guarantees the agent re-enters planning even if no semantic boundary fires.

**Roll-up: which signals ARM / CONSUME / FORCE-RE-ARM / VOID**
- **ARM** (open a bounded grant = "you may proceed through this phase"): operation-mode transitions (1), edit-batch start (2), verification start (3), working-set cluster change (6).
- **CONSUME** (a grant authorizes a bounded unit of work): edit-batch end / PostToolBatch (2), verification start consuming the prior edit grant (3).
- **FORCE EARLY RE-ARM** (risk escalation — require fresh deliberate planning before proceeding): N consecutive same-error failures (4), protected-path/destructive action (5).
- **VOID** (invalidate the grant): finalization/Stop (7), TTL/tool-count expiry (8), and failure loops (4).

**Telemetry-schema callout.** On a **minimal row** (`session_id`, `tool_name`, one collapsed `target`, `status`, `timestamp`): signals 1, 3, 4, 5, 7, 8 work fully; signal 2's batch-end needs the PostToolBatch event (Claude Code) and degrades to a mode-transition heuristic elsewhere. **Signal 6 (working-set drift) needs the richest telemetry** — it requires the target to be a *path* you can take a directory prefix of, and ideally a short history of recent prefixes (more than one row of memory). CwdChanged gives a cheap deterministic substitute on Claude Code. The behavioral-firewall result (24% benign breakage under drift) is a direct warning: keep signal 6 coarse and treat it as an arming hint, not a hard gate.

**Cross-tool feasibility.** The design is fully realizable on **Claude Code** (rich event set incl. PostToolBatch, PostToolUseFailure, CwdChanged; deny is non-bypassable). On **Codex CLI** it is only partially realizable today: PreToolUse/PostToolUse see only `Bash`, so MUTATE detection for Edit/Write/apply_patch and most of the trajectory must be reconstructed from Bash + the Stop hook, and the docs warn PreToolUse is "a useful guardrail rather than a complete enforcement boundary." For Codex, lean on its native approval-policy/sandbox layer and `requirements.toml` for the destructive-action gate, and use the Stop hook for the re-arm-on-finalize logic.

## Recommendations

**Stage 1 — Build the deterministic core on Claude Code first.** Implement the gate as a PreToolUse hook (command type) that reads/writes the per-session grant record and arms on the two highest-confidence, least-gameable signals: **READ→MUTATE** and **MUTATE→VERIFY** mode transitions (signal 1). Add the TTL/tool-count expiry backstop (signal 8) immediately — it alone guarantees periodic re-arming. Use a static tool-class map; persist the grant record to a small SQLite/JSON keyed by `session_id`. *Threshold to advance:* the gate correctly arms on real phase transitions in replayed trajectories with <5% spurious arms.

**Stage 2 — Add risk escalation and failure handling.** Wire signal 5 (protected-path/destructive) to `permissionDecision: "deny"`/`"ask"` for non-bypassable escalation, and signal 4 (N=3 consecutive same-error failures) via PostToolUseFailure to FORCE re-arm and VOID in-flight grants. Adopt a coarse error signature (exit code + collapsed target) per the Reflexion/Scheduler-Theoretic guidance. *Threshold:* retry amplification (à la Guardrails-as-Infrastructure) measurably drops without task-success collapse.

**Stage 3 — Add batch and finalization semantics.** Use PostToolBatch to CONSUME edit grants at batch end and Stop to VOID at finalization (guard against the 8-block cap via `stop_hook_active`). Add a Stop-hook working-tree scan to catch Bash-mediated edits the matcher missed.

**Stage 4 — Add working-set drift cautiously.** Only after Stages 1–3 are stable, add signal 6 as an *arming hint* using coarse directory-prefix clustering and CwdChanged. Keep it soft (arm, never hard-deny) given the documented 24% benign-breakage risk of sequence-shape enforcement under drift. *Threshold to keep it:* it improves boundary recall without raising benign-task friction above a few percent.

**Stage 5 — Port to Codex with reduced expectations.** Implement the Bash-observable subset (VERIFY/COMMIT detection, TTL, failure loops, Stop-based re-arm) via Codex hooks, and delegate the destructive-action gate to Codex's native approval-policy/sandbox + `requirements.toml`. Document explicitly that Codex PreToolUse is not a complete enforcement boundary.

**Cross-cutting:** Keep SequentialThinking optional — when a grant is armed, *offer* it; never require an agent-authored token to re-arm. Make every grant write idempotent (LangGraph's pre-interrupt idempotency rule). Log every arm/consume/force/void to a tamper-evident audit trail (the AEGIS/behavioral-firewall audit pattern).

## Caveats
- **Future-dated sources.** Many of the most on-point papers (TraceSafe 2604.07223, Guardrails-as-Infrastructure 2603.18059, AEGIS 2603.12621, the behavioral firewall 2604.26274, the Scheduler-Theoretic framework 2604.11378, the hierarchical web-agent study 2603.14248) carry 2026 arXiv IDs. I verified each resolves to a real arXiv page with consistent abstract/author metadata, but they are recent preprints (several "under review"), so treat specific numbers as preprint-grade, not peer-reviewed-final.
- **"Praetor" naming.** The behavioral-firewall system is called "Praetor" only in the secondary blog (Ilya Kabanov, theweatherreport.ai); the underlying paper (arXiv:2604.26274, Hung Dang, Van Lang University) uses an anonymized codename. Cite the paper, not the nickname.
- **Hook surfaces change.** Claude Code's hook set is large and versioned (the `if` field is v2.1.85+; PostToolBatch and others are recent); Codex hooks are explicitly experimental with Windows disabled. Pin to the version you deploy against and re-verify event names before shipping.
- **Determinism has limits.** Tool-class transitions are a *proxy* for semantic phases; they will sometimes mis-segment (e.g., a read done mid-edit for context). The design deliberately uses coarse, hard-to-game signals and an unconditional TTL backstop precisely because fine-grained semantic segmentation is not deterministically computable from a minimal telemetry row.
- **Bash escape hatch.** On both CLIs, an agent can perform file mutations through Bash, evading Edit/Write-class detection. This is a known, documented gap; the working-tree-scan mitigation is partial, not complete.
